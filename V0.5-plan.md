# V0.5 Slice Plan: Meta Ad Account Connection

**Slice:** V0.5 — Meta Ad Account Connection
**Parts:** A10
**Demo:** User initiates Meta OAuth, connects an Ad Account, sees it listed in Settings, can disconnect it
**Stack:** React + Vite, Hono (Node), Neon Postgres, Drizzle ORM, Meta Marketing API (OAuth), Portless — full TypeScript
**Requirement:** R13 — Users can connect specific Meta Ad Accounts via OAuth; tool uses these credentials to act on their behalf. User selects which ad account per execution.

### Step 2: Database schema — Meta Ad Accounts table

Add a new table to `packages/db/src/schema.ts`:

**`public.meta_ad_accounts`**

| Column | Type | Notes |
|--------|------|-------|
| id | uuid, PK | default gen_random_uuid() |
| company_id | uuid, NOT NULL | FK → companies.id — scopes account to tenant |
| connected_by_user_id | text, NOT NULL | The user_id (from auth) who connected this account |

| meta_account_name | text, NOT NULL | Human-readable name of the ad account |
| encrypted_access_token | text, NOT NULL | AES-256-GCM encrypted long-lived access token |
| token_iv | text, NOT NULL | Initialization vector for the encrypted token |
| token_expires_at | timestamptz, nullable | When the token expires (null if non-expiring) |
| token_status | text, NOT NULL, default 'valid' | 'valid', 'expired', or 'error' |
| created_at | timestamptz | default now() |
| updated_at | timestamptz | default now() |

**Unique constraint:** `(company_id, meta_account_id)` — prevent connecting the same ad account twice within a company.

### Step 3: Token encryption utility

Create `apps/api/src/lib/crypto.ts`:

- `encrypt(plaintext: string): { ciphertext: string; iv: string }` — AES-256-GCM using `TOKEN_ENCRYPTION_KEY` from env
- `decrypt(ciphertext: string, iv: string): string` — reverse
- Uses Node.js built-in `crypto` module (no external dependencies)
- Key is read from `TOKEN_ENCRYPTION_KEY` env var (32-byte hex string)
- Throws clear error if env var is missing or malformed

This utility is used to encrypt Meta access tokens before storing them and decrypt them before making API calls.

### Step 4: Meta OAuth routes — auth-url + callback

Create `apps/api/src/routes/meta.ts` with two routes:

**`GET /api/meta/auth-url`** (protected — requires `authMiddleware`)
- Generates a random `state` parameter and stores it temporarily (in-memory map with TTL, keyed by state value, storing `{ userId, companyId }`)
- Constructs Meta OAuth URL:
  ```
  https://www.facebook.com/v21.0/dialog/oauth
    ?client_id={META_APP_ID}
    &redirect_uri={META_OAUTH_REDIRECT_URI}
    &state={state}
    &scope=ads_management,ads_read,business_management
    &response_type=code
  ```
- Returns `{ url }` per contract

**`GET /api/meta/callback`** (public route — Meta redirects here)
- Receives `code` and `state` query params
- Validates `state` against the in-memory map (prevents CSRF). Returns error if state not found or expired.
- Exchanges `code` for a short-lived token via Meta's token endpoint:
  ```
  GET https://graph.facebook.com/v21.0/oauth/access_token
    ?client_id={META_APP_ID}
    &redirect_uri={META_OAUTH_REDIRECT_URI}
    &client_secret={META_APP_SECRET}
    &code={code}
  ```
- Exchanges short-lived token for a long-lived token:
  ```
  GET https://graph.facebook.com/v21.0/oauth/access_token
    ?grant_type=fb_exchange_token
    &client_id={META_APP_ID}
    &client_secret={META_APP_SECRET}
    &fb_exchange_token={short_lived_token}
  ```
- Fetches the user's ad accounts from Meta:
  ```
  GET https://graph.facebook.com/v21.0/me/adaccounts
    ?access_token={long_lived_token}
    &fields=account_id,name,account_status
  ```
- Redirects to frontend: `{FRONTEND_URL}/settings/meta-accounts?status=select&meta_user_id={id}` with the ad accounts list stored in a temporary server-side session/store keyed by the state
- (The actual ad account selection and storage happens in Step 5)

Note: The callback is a GET endpoint because Meta redirects the browser to it. It should redirect to the frontend after processing, not return JSON.

**Agent:** Backend
**Branch:** `v0.5/step-4-meta-oauth-routes`
**Done when:** `GET /api/meta/auth-url` returns a valid Meta OAuth URL (manually testable in browser). The callback endpoint can exchange a code for tokens (testable with a real Meta test app or mocked in integration tests).

### Step 5: Meta account selection + storage endpoints

After the OAuth callback redirects to the frontend with a temporary session, the frontend needs to let the user select which ad account(s) to connect.

**`GET /api/meta/pending-accounts`** (protected)
- Returns the list of ad accounts fetched during the OAuth callback, stored temporarily server-side keyed by `meta_user_id`
- Response: `{ accounts: [{ id, name, status }] }`

**`POST /api/meta/accounts`** (protected)
- Body: `{ metaUserId: string, selectedAccountIds: string[] }`
- For each selected ad account:
  - Encrypts the long-lived access token using `crypto.encrypt()`
  - Inserts into `meta_ad_accounts` with `company_id` from auth context, `connected_by_user_id` from auth context
  - Skips if `(company_id, meta_account_id)` already exists (returns warning)
- Clears the temporary session data
- Returns: `{ connected: number, skipped: number }`

**`GET /api/meta/accounts`** (protected)
- Lists all connected meta ad accounts for the authenticated user's company
- Queries `meta_ad_accounts WHERE company_id = auth.companyId`
- Returns data matching `ListMetaAccountsResponse` contract (does NOT return tokens)

**`DELETE /api/meta/accounts/:id`** (protected)
- Deletes a connected ad account by its `meta_ad_accounts.id`
- Validates that the account belongs to `auth.companyId`
- Returns `{ success: true }`

**Branch:** `v0.5/step-5-meta-account-crud`
**Done when:** Full CRUD flow works via curl/Postman: list accounts (empty), connect accounts (via POST with test data), list again (shows them), delete one, list again (gone). All endpoints enforce auth + company scoping.

### Step 6: Token refresh background job

Create `apps/api/src/jobs/meta-token-refresh.ts`:

- Queries `meta_ad_accounts` for tokens expiring within 7 days (`token_expires_at < NOW() + INTERVAL '7 days'` AND `token_status = 'valid'`)
- For each:
  - Decrypts the stored access token
  - Calls Meta's token refresh endpoint:
    ```
    GET https://graph.facebook.com/v21.0/oauth/access_token
      ?grant_type=fb_exchange_token
      &client_id={META_APP_ID}
      &client_secret={META_APP_SECRET}
      &fb_exchange_token={current_token}
    ```
  - On success: encrypts new token, updates `encrypted_access_token`, `token_iv`, `token_expires_at`, `updated_at`
  - On failure: sets `token_status = 'error'`, logs the error
- Exposed as a function that can be called:
  - On app startup (runs once)
  - On a `setInterval` (every 12 hours)
  - Via `POST /api/meta/accounts/:id/refresh` (manual trigger, protected, super_admin or the user who connected)

**Agent:** Backend
**Branch:** `v0.5/step-6-token-refresh`
**Done when:** The refresh job correctly identifies expiring tokens. Manual refresh endpoint works. Token status transitions are correct (valid → valid on success, valid → error on failure).

## Security Considerations

1. **Token encryption at rest:** All Meta access tokens are encrypted with AES-256-GCM before storage. The encryption key (`TOKEN_ENCRYPTION_KEY`) is a 32-byte hex string stored only in environment variables, never in code or version control.
2. **CSRF protection on OAuth:** The `state` parameter in the OAuth flow is a cryptographically random value stored server-side with a short TTL (10 minutes). The callback validates the state before proceeding.
3. **Company scoping:** All `meta_ad_accounts` queries filter by `company_id` from the authenticated user's context. A user in Company A can never see or manipulate Company B's connected accounts.
4. **Minimal token scope:** OAuth requests only the scopes needed: `ads_management`, `ads_read`, `business_management`.
5. **No tokens in API responses:** The `GET /api/meta/accounts` endpoint never returns access tokens, IVs, or any token-related fields. Only metadata.
6. **Token refresh proactivity:** Tokens are refreshed 7 days before expiry to avoid disruption. Failed refreshes are flagged with `token_status = 'error'`.
7. **App Secret protection:** `META_APP_SECRET` is only used server-side, never exposed to the frontend.

## Out of Scope for V0.5

- Ad account selector per campaign execution (will be built in V1+ when campaign execution exists)
- Webhook-based token invalidation notifications from Meta
- Multiple Meta apps / multiple platform connections (only Meta for MVP)
- Rate limiting on Meta API calls
- Batch operations on ad accounts
- Token migration or rotation tooling
